## 1. 시작하며

2026년 1학기 동안 통계학과 학회 BAF에서 금융 데이터 분석 프로젝트를 진행했다. 평소 금융 데이터 분석을 해보고 싶다는 생각은 있었지만, 실제로 금융 데이터를 직접 수집하고 분석해본 경험은 거의 없었다. 특히 스테이블코인이나 디페깅이라는 개념도 프로젝트를 시작하면서 처음 제대로 접했다.

처음에는 “스테이블코인의 가격 이탈을 데이터로 미리 예측할 수 있을까?”라는 질문 자체가 낯설었다. 금융 데이터는 변동성이 크고, 디페깅처럼 자주 발생하지 않는 사건을 예측하는 문제는 단순한 가격 예측보다 훨씬 까다로울 것이라고 생각했다.

그럼에도 불구하고 이 주제를 선택한 이유는 명확했다. 가격, 거래량, 거시경제 변수, 뉴스 데이터, 온체인 데이터 등 여러 종류의 데이터를 결합할 수 있었고, 단순 예측을 넘어 실제 리스크를 사전에 탐지하는 조기경보 모델이라는 점에서 실무적으로도 의미가 있다고 판단했기 때문이다.

이번 프로젝트는 나에게 금융 데이터 분석을 처음으로 깊게 경험한 프로젝트였다. 동시에 데이터 분석에서 중요한 것은 단순히 모델을 많이 돌리는 것이 아니라, 예측하고자 하는 사건을 어떻게 정의할 것인지, 불균형한 데이터를 어떻게 다룰 것인지, 실험 결과를 어떻게 해석하고 받아들일 것인지를 고민하는 과정이라는 점을 배운 프로젝트이기도 했다.

## 2. 프로젝트 개요

이번 프로젝트의 주제는 **USDC와 DAI 스테이블코인의 디페깅 조기경보 모델 구축**이었다.

스테이블코인은 일반적으로 1달러에 가까운 가치를 유지하도록 설계된 가상자산이다. 하지만 시장 충격, 유동성 위기, 담보 구조 문제, 거시경제 변화 등에 따라 기준 가격에서 이탈하는 디페깅이 발생할 수 있다. 실제로 과거 UST 붕괴, USDC의 SVB 사태, DAI의 연쇄 디페깅 사례처럼 디페깅은 단순한 가격 변동을 넘어 시장 전반의 리스크로 확산될 수 있다.

본 프로젝트에서는 이러한 디페깅 위험을 사후적으로 설명하는 데 그치지 않고, 발생 가능성을 사전에 탐지할 수 있는 머신러닝 기반 조기경보 체계를 구축하는 것을 목표로 했다.

### 프로젝트 정보

* 기간: 2026년 1학기, 약 3개월
* 활동: BAF 금융팀 프로젝트
* 분석 대상: USDC, DAI
* 주요 데이터: 가격, 거래량, 비트코인 가격, 금리, 거시경제 지표, 뉴스 데이터, 온체인 데이터
* 주요 모델: XGBoost, LightGBM, Random Forest, SVM
* 주요 평가 지표: F2-score, Recall

이번 프로젝트의 핵심 목표는 단순히 “디페깅 여부를 맞히는 모델”을 만드는 것이 아니었다. 디페깅은 발생 빈도가 낮지만, 한 번 발생했을 때 시장에 미치는 영향이 크기 때문에 실제 위험을 놓치지 않는 것이 중요했다. 따라서 전체 정확도보다 Recall과 F2-score를 중심으로 모델을 평가했다.

특히 F2-score를 주요 지표로 설정한 이유는 디페깅 조기경보 문제에서 실제 위험을 놓치는 것, 즉 False Negative를 줄이는 것이 중요하다고 판단했기 때문이다.

## 3. 주제 선정 과정

프로젝트 초반에는 금융 데이터 분석이라는 큰 방향만 정해져 있었고, 구체적인 주제는 팀원들과 논의하며 좁혀갔다. 나는 금융 데이터 분석을 제대로 경험해보고 싶었고, 팀원들과의 논의 과정에서 최근 스테이블코인과 관련된 시장 이슈가 자주 언급된다는 점을 알게 되었다.

스테이블코인 디페깅은 단순한 가격 예측과 달리, 시장 안정성이나 유동성 위험과 연결되는 문제였다. 또한 가격과 거래량뿐 아니라 거시경제 변수, 뉴스 데이터, 온체인 데이터까지 함께 사용할 수 있다는 점에서 분석 주제로 확장성이 있었다.

다만 처음부터 확신이 있었던 것은 아니다. 가장 큰 고민은 “디페깅이라는 사건이 실제로 예측 가능한가?”였다. 디페깅은 자주 발생하는 사건이 아니기 때문에, 모델이 충분히 학습할 수 있을 만큼 양성 데이터가 존재하는지부터 불확실했다. 또 디페깅을 어떻게 정의하느냐에 따라 타깃 값이 달라지고, 결국 모델 결과도 크게 달라질 수 있었다.

이 고민 때문에 프로젝트 초반에는 타깃 정의에 많은 시간을 썼다. 단순히 종가가 1달러에서 벗어났는지를 보는 방식은 일중 변동을 충분히 반영하지 못한다고 판단했고, 최종적으로 고가·저가·종가의 평균인 Typical Price를 활용했다. 여기에 단기적인 가격 노이즈를 줄이기 위해 2-of-3 필터를 적용하여, 일시적 이탈이 아닌 지속적인 이탈을 디페깅으로 정의했다.

## 4. 내가 맡은 역할

이번 프로젝트에서 나는 뉴스 기반 텍스트 분석과 파생변수 생성, 그리고 해당 변수들이 모델 성능에 실제로 기여하는지 검증하는 과정에 중점적으로 참여했다. 또한 타깃 정의, 변수 설계, 불균형 처리, 모델 평가 방식에 대한 팀 논의에도 함께 참여했다.

### My Contribution

* 뉴스 기반 텍스트 파생변수 생성
* 리스크/긍정 키워드 기반 변수 실험
* LDA 토픽 변수 생성
* FinBERT 기반 임베딩 변수 실험
* 텍스트 변수 포함 전후 F2-score 비교
* 텍스트 변수가 최종 성능 개선에 기여하지 못한 이유 정리
* 디페깅 정의, 불균형 처리, 모델 평가 방식에 대한 팀 논의 참여

구체적으로는 뉴스 데이터에서 리스크 관련 키워드와 긍정 키워드를 추출하고, LDA 기반 토픽 변수와 FinBERT 기반 임베딩 변수를 생성하는 실험을 진행했다. 텍스트 데이터는 금융시장의 심리나 위험 신호를 반영할 수 있을 것이라고 기대했기 때문에, 가격·거래량·거시경제 변수만으로는 포착하기 어려운 정보를 보완할 수 있을 것이라 생각했다.

하지만 실험 결과, 텍스트 변수는 최종 모델의 F2-score를 개선하지 못했다. 처음에는 만든 변수가 성능을 높이지 못했다는 점이 아쉬웠지만, 프로젝트를 돌아보면 이 과정이 오히려 중요한 경험이었다. 데이터 분석에서는 새로운 변수를 많이 만드는 것보다, 그 변수가 실제로 예측 성능에 기여하는지 검증하고 필요하다면 제외하는 판단이 중요하다는 점을 배웠기 때문이다.

또한 디페깅 데이터는 양성 클래스가 매우 적은 불균형 데이터였기 때문에, 모델 성능을 단순 정확도로 판단하기 어려웠다. 정상 구간이 대부분인 데이터에서는 Accuracy가 높아도 실제 디페깅을 제대로 포착하지 못할 수 있기 때문이다. 이 문제를 해결하기 위해 Recall과 F2-score를 중심으로 모델 성능을 비교했고, 최종적으로 USDC는 WeightOnly와 XGBoost, DAI는 ADASYN과 XGBoost 조합이 가장 적합하다고 판단했다.

이번 프로젝트를 통해 내가 가장 크게 느낀 점은, 분석 과정에서 중요한 의사결정은 모델 선택보다 앞단에 있다는 것이었다. 무엇을 예측할 것인지 정의하고, 어떤 데이터를 사용할지 선택하고, 어떤 지표로 성능을 평가할지 정하는 과정이 모델링만큼 중요했다.

## 5. 분석 설계 과정

이번 프로젝트에서 가장 먼저 부딪힌 문제는 “디페깅을 어떻게 정의할 것인가”였다. 디페깅이라는 단어는 직관적으로는 1달러에서 벗어나는 현상을 의미하지만, 실제 데이터에서 이를 라벨로 만들려면 기준이 필요했다.

처음에는 외부 자료를 참고해 여러 동적 임계값 방식을 검토했지만, 하루 동안만 가격이 튀는 경우까지 디페깅으로 잡히는 문제가 있었다. 이런 단기 변동은 실제 위험 신호라기보다 노이즈에 가까울 수 있다고 판단했다.

이 문제를 해결하기 위해 논문과 관련 자료를 찾아보며 팀원들과 디페깅 정의를 다시 논의했다. 최종적으로는 고가, 저가, 종가의 평균인 Typical Price를 활용하고, 기준 가격 1달러에서 ±1% 이상 이탈한 경우를 디페깅 후보로 보았다. 여기에 3일 중 2일 이상 이탈이 관측되는 경우만 디페깅으로 판단하는 2-of-3 필터를 적용했다.

이 방식은 하루만 튀는 일시적 가격 변동을 줄이고, 비교적 지속적인 가격 이탈을 디페깅으로 잡기 위한 선택이었다. 이 과정에서 타깃 정의가 단순한 전처리 단계가 아니라, 모델 전체의 성능과 해석을 좌우하는 핵심 의사결정이라는 점을 느꼈다.

*그림 2. Typical Price와 2-of-3 필터 기반 디페깅 타깃 정의*

분석 파이프라인은 크게 7단계로 구성했다.

1. 디페깅 타깃 정의
2. 가격, 거래량, 거시경제, 온체인 데이터 기반 피처 엔지니어링
3. 뉴스 기반 텍스트 파생변수 생성
4. 불균형 데이터 처리
5. XGBoost, LightGBM, Random Forest, SVM 모델 비교
6. SHAP 기반 변수 중요도 해석
7. 정상, 주의, 경보로 나누는 3단계 조기경보 시스템 구성

이 중 내가 가장 의미 있게 관여한 부분은 텍스트 파생변수 실험과, 디페깅 정의 및 불균형 처리에 대한 논의였다. 특히 뉴스 데이터가 시장 심리를 반영할 수 있을 것이라고 생각했기 때문에, 텍스트 변수가 기존 가격·거래량 변수만으로는 설명하기 어려운 위험 신호를 보완할 수 있을지 확인하고 싶었다.

## 6. 주요 실험과 결과

### 6.1 텍스트 파생변수 실험

뉴스 데이터는 금융시장 위험을 반영할 수 있는 중요한 정보원이라고 생각했다. 그래서 뉴스 기사에서 리스크 관련 키워드와 긍정 키워드를 추출하고, LDA 토픽 모델링과 FinBERT 기반 임베딩 변수를 활용해 텍스트 파생변수를 만들었다.

처음에는 텍스트 변수를 추가하면 모델 성능이 개선될 것이라고 기대했다. 스테이블코인 디페깅은 단순한 가격 움직임만으로 발생하는 것이 아니라, 시장 불안, 담보 리스크, 규제 이슈, 유동성 불안 등과도 연결될 수 있기 때문이다. 뉴스 데이터가 이런 정보를 일부 반영할 수 있다면, 조기경보 모델에 도움이 될 것이라고 생각했다.

하지만 결과는 예상과 달랐다. 텍스트 변수를 포함했을 때 F2-score가 개선되지 않았고, 오히려 일부 성능이 하락했다. 처음에는 당황스러웠다. 공을 들여 만든 변수가 성능 개선에 기여하지 못했다는 점이 아쉬웠기 때문이다.

하지만 이 결과를 받아들이면서 중요한 점을 배웠다. 분석에서 중요한 것은 내가 만든 변수를 반드시 모델에 넣는 것이 아니라, 그 변수가 실제로 예측 성능에 기여하는지 검증하는 것이다. 텍스트 변수는 직관적으로는 의미 있어 보였지만, 실제 데이터에서는 양성 샘플이 부족하고 신호보다 노이즈로 작용할 가능성이 있었다. 결국 최종 모델에서는 텍스트 변수를 제외하는 방향으로 정리했다.

이 경험을 통해 “좋아 보이는 변수”와 “실제로 모델 성능을 개선하는 변수”는 다를 수 있다는 점을 배웠다.

---

### 6.2 불균형 데이터 처리

두 번째로 어려웠던 부분은 불균형 데이터 처리였다. 디페깅은 정상 상태에 비해 발생 빈도가 매우 낮은 사건이기 때문에, 전체 데이터에서 양성 클래스가 차지하는 비율이 작았다. 이런 상황에서는 단순히 Accuracy가 높다고 해서 좋은 모델이라고 보기 어렵다. 대부분을 정상으로 예측해도 정확도는 높게 나올 수 있기 때문이다.

![디페깅 라벨 분포와 이벤트 집중](https://raw.githubusercontent.com/hyunjun216/stablecoin-depegging-early-warning/main/figures/velog/03_depeg_label_distribution.png)

*그림 3. 디페깅 라벨 분포와 2020년 이벤트 집중*

그래서 모델 성능을 평가할 때 Accuracy보다 Recall과 F2-score를 중요하게 보았다. 디페깅 조기경보 모델에서는 실제 위험을 놓치지 않는 것이 중요하므로, False Negative를 줄이는 방향의 평가 지표가 필요하다고 판단했다.

불균형 처리 방식도 하나로 고정하지 않고 여러 방법을 비교했다.

![불균형 처리별 F2-score 비교](https://raw.githubusercontent.com/hyunjun216/stablecoin-depegging-early-warning/main/figures/velog/04_imbalance_f2_comparison.png)

*그림 4. USDC와 DAI의 불균형 처리별 F2-score 비교*

USDC는 양성 데이터가 매우 적었기 때문에 합성 데이터를 만드는 방식이 오히려 과적합 위험을 키울 수 있다고 보았고, 최종적으로 소수 클래스에 가중치를 부여하는 WeightOnly 방식이 가장 좋은 결과를 보였다. 반면 DAI는 상대적으로 양성 샘플이 더 많아 ADASYN을 활용한 방식이 더 적합했다.

이 결과를 통해 불균형 데이터 처리는 “항상 SMOTE를 쓰면 된다”처럼 단순하게 접근할 수 없다는 점을 배웠다. 데이터의 양성 클래스 수, 분포, 모델 특성에 따라 적절한 방식이 달라질 수 있었다.

---

### 6.3 모델 비교와 XGBoost 선택

불균형 처리 방식을 정한 뒤에는 XGBoost, LightGBM, Random Forest, SVM을 비교했다.

![모델별 F2-score 비교](https://raw.githubusercontent.com/hyunjun216/stablecoin-depegging-early-warning/main/figures/velog/05_model_comparison_f2.png)

*그림 5. 모델별 F2-score 비교*

최종적으로 두 코인 모두 XGBoost가 가장 좋은 F2-score를 보였고, USDC는 WeightOnly + XGBoost, DAI는 ADASYN + XGBoost 조합을 선택했다.

| Coin | Imbalance Handling | Final Model | F2-score | Recall |
| ---- | ------------------ | ----------- | -------: | -----: |
| USDC | WeightOnly         | XGBoost     |    0.906 |  0.900 |
| DAI  | ADASYN             | XGBoost     |    0.878 |  0.905 |

XGBoost를 선택한 이유는 단순히 성능이 높았기 때문만은 아니었다. 비선형 관계와 변수 간 상호작용을 포착하는 데 강점이 있고, SHAP 기반 해석과도 연결하기 좋다는 점이 있었다. 디페깅 조기경보 모델은 단순히 예측 결과만 내는 것이 아니라, 어떤 변수가 위험 신호로 작용했는지 해석하는 것도 중요했기 때문에 모델 해석 가능성도 함께 고려했다.

최종 결과에서는 USDC와 DAI 모두 Recall 90% 이상을 달성했고, F2-score도 목표 기준을 넘었다.

![SHAP Top 변수 비교](https://raw.githubusercontent.com/hyunjun216/stablecoin-depegging-early-warning/main/figures/velog/06_shap_top_features.png)

*그림 6. USDC와 DAI의 SHAP Top 변수 비교*

*그림 7. 정상·주의·경보로 구성한 3단계 조기경보 시스템*

## 7. 가장 어려웠던 점

이번 프로젝트에서 가장 어려웠던 점은 모델링 자체보다 “어떤 방식이 올바른 검증인가”를 판단하는 것이었다.

첫 번째 어려움은 디페깅 정의였다. 디페깅을 어떻게 정의하느냐에 따라 양성 라벨 수가 달라지고, 모델 성능도 크게 달라질 수 있었다. 처음에는 하루만 가격이 튀어도 디페깅으로 잡히는 문제가 있었고, 이는 실제 위험 신호라기보다 단기 노이즈에 가까울 수 있었다. 그래서 Typical Price와 2-of-3 필터를 적용해 지속적인 가격 이탈을 중심으로 라벨을 정의했다.

두 번째 어려움은 불균형 데이터였다. 디페깅은 자주 발생하지 않는 사건이기 때문에, 정상 데이터가 압도적으로 많았다. 이 상황에서 단순 정확도만 보면 모델이 실제 위험을 잘 잡는지 판단하기 어려웠다. 그래서 Recall과 F2-score를 중심으로 성능을 해석했고, 코인별로 다른 불균형 처리 방식을 적용했다.

세 번째 어려움은 미래정보 사용 가능성과 시계열 검증 방식에 대한 고민이었다. 조기경보 모델에서는 예측 시점 이후의 정보가 변수에 섞이면 실제보다 성능이 높게 보일 수 있다. 따라서 단순히 높은 성능을 얻는 것보다, 예측 시점에서 사용할 수 있는 정보만으로 평가하는 구조가 중요하다고 느꼈다.

최종 정리 과정에서는 단순히 “성능이 좋은 모델”을 만드는 것을 넘어, 시간적 편중과 검증 방식의 한계를 함께 명시했다. 특히 디페깅 이벤트가 특정 시점에 집중되어 있었기 때문에 일반적인 시계열 분할만으로는 안정적인 평가가 어려웠고, 실제 운용에서는 rolling window 방식의 재학습이 필요하다고 보았다.

이 과정을 통해 데이터 분석에서 가장 위험한 것은 낮은 성능이 아니라, 잘못된 검증으로 높은 성능을 얻는 것일 수 있다는 점을 배웠다.

---

## 8. 프로젝트를 통해 배운 점

이번 프로젝트를 통해 가장 크게 배운 것은 데이터 분석에서 “문제 정의”와 “검증 설계”가 모델링만큼 중요하다는 점이다.

처음에는 모델을 잘 선택하고 성능을 높이는 것이 핵심이라고 생각했다. 하지만 프로젝트를 진행하면서 디페깅을 어떻게 정의할지, 단기 노이즈를 어떻게 제거할지, 양성 데이터가 적은 상황에서 어떤 지표를 볼지, 미래정보가 섞이지 않도록 어떻게 검증할지를 고민하는 과정이 훨씬 중요하다는 것을 알게 되었다.

또한 예상과 다른 결과를 받아들이는 태도도 배웠다. 텍스트 변수는 처음에 성능 개선을 기대했던 부분이었지만, 실제 결과는 그렇지 않았다. 과거의 나라면 “왜 성능이 안 오르지?”에만 집중했을 것 같다. 하지만 이번 프로젝트에서는 실제 데이터가 보여주는 결과를 받아들이고, 성능에 기여하지 못하는 변수는 제외하는 것이 더 합리적이라는 점을 배웠다.

마지막으로, 금융 데이터 분석은 단순한 예측 문제가 아니라 해석과 리스크 관리의 문제라는 점을 느꼈다. 디페깅 조기경보 모델은 특정 날짜의 라벨을 맞히는 것에서 끝나지 않는다. 어떤 변수가 위험 신호로 작용했는지, 어떤 상황에서 경보를 낼 것인지, 오경보와 미탐지 중 무엇을 더 중요하게 볼 것인지까지 함께 고민해야 한다.

이번 프로젝트는 완벽한 결과물이라기보다, 금융 데이터 분석을 처음으로 깊게 경험하고 데이터 분석의 기준을 배운 프로젝트였다.

---

## 9. 아쉬웠던 점

프로젝트를 끝내고 가장 아쉬웠던 점은 시계열 데이터 검증을 충분히 다루지 못했다는 점이다.

처음에는 모델 성능을 높이는 데 집중했지만, 프로젝트를 진행할수록 금융 데이터에서는 “시간 순서”가 매우 중요하다는 것을 느꼈다. 일반적인 분류 문제처럼 데이터를 무작위로 나누어 학습하고 평가하면 안정적인 성능 지표를 얻을 수는 있지만, 실제 예측 상황과는 차이가 생길 수 있다. 특히 조기경보 모델에서는 예측 시점 이후의 정보가 학습 과정에 섞이지 않아야 하며, 미래정보 사용 가능성을 계속 점검해야 한다.

이 프로젝트를 진행하며 가장 크게 고민한 부분은 미래정보 사용 가능성과 시계열 검증 방식이었다. 조기경보 모델에서는 예측 시점 이후의 정보가 변수에 섞이면 실제보다 성능이 높게 보일 수 있기 때문에, 단순히 높은 성능을 얻는 것보다 예측 시점에서 사용할 수 있는 정보만으로 평가하는 구조가 중요하다고 느꼈다.

최종적으로는 디페깅 이벤트가 특정 시점에 집중되어 일반적인 시계열 분할만으로는 안정적인 평가가 어렵다는 한계를 확인했고, 향후 rolling window 방식의 검증이 필요하다고 보았다.

![시계열 검증 한계와 Rolling Window 필요성](https://raw.githubusercontent.com/hyunjun216/stablecoin-depegging-early-warning/main/figures/velog/08_time_series_validation_limitation.png)

*그림 8. 시계열 검증 한계와 Rolling Window 필요성*

또한 일별 데이터를 중심으로 분석했기 때문에 디페깅 발생 직전의 미세한 변화를 충분히 포착하기에는 한계가 있었다. 향후에는 hourly 데이터처럼 더 세밀한 시계열 데이터를 활용해, 디페깅 발생 전후의 변화를 더 정교하게 살펴보고 싶다.

이번 프로젝트는 결과 자체로도 의미가 있었지만, 동시에 내가 아직 시계열 데이터 검증과 금융 데이터 구조를 충분히 이해하지 못하고 있다는 사실을 확인한 프로젝트이기도 했다. 이 점은 아쉬움이지만, 다음 프로젝트에서 무엇을 보완해야 하는지 분명하게 알려준 부분이기도 하다.

## 10. 앞으로 보완하고 싶은 점

다시 이 프로젝트를 진행한다면 가장 먼저 시계열 검증 방식을 개선하고 싶다. 단순히 전체 데이터를 섞어 학습과 평가를 나누는 것이 아니라, 특정 시점까지의 데이터로 학습하고 이후 시점의 데이터를 예측하는 방식으로 검증해야 실제 조기경보 모델에 더 가까워질 수 있다고 생각한다.

구체적으로는 rolling window나 expanding window validation을 적용해보고 싶다. 예를 들어 과거 일정 기간의 데이터로 모델을 학습한 뒤 다음 기간을 예측하고, 다시 학습 구간을 이동시키는 방식이다. 이렇게 하면 모델이 실제 운용 환경에서 새로운 데이터가 들어올 때 얼마나 안정적으로 작동하는지 더 현실적으로 확인할 수 있을 것이다.

또한 금융 데이터 자체에 대한 이해도 더 필요하다고 느꼈다. 스테이블코인 디페깅은 단순히 가격이 1달러에서 벗어나는 현상이 아니라, 담보 구조, 유동성, 금리, 달러 인덱스, 시장 심리, 온체인 흐름 등이 함께 작용하는 문제였다. 앞으로는 금융시장 구조와 가상자산 시장의 메커니즘을 더 공부해, 변수를 만들 때도 단순히 많이 넣는 것이 아니라 왜 필요한 변수인지 설명할 수 있는 수준으로 발전시키고 싶다.

NLP도 보완하고 싶은 부분이다. 이번 프로젝트에서는 뉴스 기반 키워드 변수, LDA 토픽 변수, FinBERT 임베딩 변수를 실험했지만, 최종 성능 개선에는 기여하지 못했다. 다음에는 뉴스의 시점 정렬, 기사 중복 제거, 금융 이벤트별 문맥 분류, 감성 점수의 시차 변수화 등을 더 정교하게 설계해보고 싶다. 텍스트 데이터가 실제 금융 리스크 탐지에 의미 있는 신호가 되기 위해서는 단순히 변수를 만드는 것보다, 어떤 시점의 어떤 정보가 예측에 사용 가능한지까지 함께 고려해야 한다는 점을 배웠다.

마지막으로 Python 구현력과 모델 원리에 대한 이해도 더 보완하고 싶다. 이번 프로젝트에서는 여러 기법을 활용했지만, 모든 과정을 완전히 내 것으로 설명할 수 있을 만큼 깊이 이해했다고 보기는 어렵다. 앞으로는 모델을 실행하는 데 그치지 않고, 왜 이 모델을 선택했는지, 어떤 지표를 봐야 하는지, 결과가 왜 그렇게 나왔는지를 더 명확히 설명할 수 있는 분석가가 되고 싶다.

## 11. 마무리하며

이번 프로젝트는 많이 헤맸지만, 그래서 더 많이 배운 프로젝트였다.

처음에는 금융 데이터 분석이라는 분야가 막연하게 흥미로웠고, 스테이블코인과 디페깅이라는 개념도 낯설었다. 하지만 프로젝트를 진행하면서 하나의 예측 모델을 만들기 위해서는 데이터 수집, 타깃 정의, 노이즈 제거, 불균형 처리, 성능 지표 선택, 검증 방식, 결과 해석까지 많은 의사결정이 필요하다는 것을 알게 되었다.

특히 이번 프로젝트를 통해 배운 가장 큰 점은 “성능이 높은 모델”보다 “검증 가능한 모델”이 중요하다는 것이다. F2-score나 Recall 같은 지표가 높게 나오는 것도 중요하지만, 그 성능이 어떤 검증 구조에서 나온 것인지, 실제 예측 상황에서도 의미가 있는지 따져보는 과정이 더 중요했다.

또한 예상과 다른 결과를 받아들이는 태도도 배웠다. 텍스트 변수는 처음에 성능 개선을 기대했던 부분이었지만, 실제 결과는 그렇지 않았다. 하지만 그 결과를 통해 변수를 만드는 것만큼이나, 변수의 기여도를 검증하고 필요할 때 제외하는 판단이 중요하다는 점을 배웠다.

이번 프로젝트는 완벽한 결과물은 아니었다. 시계열 검증, 금융 데이터 이해, NLP 활용 방식, 코드 구현력 모두에서 보완할 부분이 많았다. 하지만 그만큼 다음 프로젝트에서 무엇을 더 공부하고 개선해야 하는지 분명하게 알게 되었다.

그래서 이 프로젝트는 나에게 단순한 학회 프로젝트가 아니라, 금융 데이터 분석을 본격적으로 공부하기 위한 출발점이었다. 많이 헤맸지만, 그 과정에서 데이터 분석을 어떻게 바라봐야 하는지 조금은 알게 된 프로젝트로 기억될 것 같다.
